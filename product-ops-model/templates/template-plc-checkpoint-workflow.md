## 2) Core checkpoints (all tracks)

### Checkpoint 0: Opportunity clarity
- Required:
  - [ ] Opportunity statement complete.
  - [ ] Customer segment and impact defined.
  - [ ] KPI baseline and target set.

### Checkpoint 1: Discovery integrity
- Required:
  - [ ] Evidence recency check passes (last 14 days for active work).
  - [ ] Top assumptions documented.
  - [ ] At least one assumption test designed.

### Checkpoint 2: Commitment readiness
- Required:
  - [ ] Appetite and scope boundaries explicit.
  - [ ] Out-of-scope declared.
  - [ ] Dependencies and risks reviewed.

### Checkpoint 3: Post-launch learning
- Required:
  - [ ] Outcome performance reviewed.
  - [ ] Decision made (continue/iterate/pause/stop).
  - [ ] Learnings captured and shared.

## 3) UI-heavy track checkpoints

### UI-A: Design system alignment
- [ ] Existing component inventory checked first.
- [ ] Reuse decision documented.
- [ ] New component rationale includes generic reuse potential.

### UI-B: Handoff readiness
- [ ] Design intent documented with behavior states.
- [ ] Annotated handoff notes prepared.
- [ ] Accessibility and UX guardrails included.

### UI-C: Front-end quality
- [ ] Visual consistency check.
- [ ] Interaction and state coverage check.
- [ ] Regression/edge-case coverage defined.

## 4) Backend-heavy track checkpoints

### BE-A: Architecture alignment
- [ ] Service boundary and data flow documented.
- [ ] Existing backend patterns reviewed.
- [ ] Build-vs-reuse decision recorded.

### BE-B: Reliability and integration
- [ ] Failure modes and rollback approach defined.
- [ ] Observability plan in place (logs/metrics/alerts).
- [ ] Contract/API compatibility reviewed.

### BE-C: Operational readiness
- [ ] Runbook/support ownership assigned.
- [ ] Performance and scale guardrails set.
- [ ] Security/privacy considerations checked.
