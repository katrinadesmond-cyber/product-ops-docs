# PLC Change Matrix (from interview insights)

Source interviews:
- Karina x Ashwini (2026-03-09)
- Karina x Kai (2026-03-06)

| PLC stage | What's changing | Cursor-enabled behaviors | New risks identified | Opportunities |
|---|---|---|---|---|
| Discovery & Define | Discovery is no longer a long upfront phase; it is blended with early implementation and fast artifact generation. | Teams prompt Cursor/ChatGPT to draft data models, high-level specs, and architecture flows from partial inputs. | Building with incomplete product context creates "shooting in the dark," causing downstream rework. | Define a minimum discovery input pack (problem, constraints, acceptance criteria, key references) before major build starts. |
| Design | Design is increasingly co-evolving with build instead of being fully complete before coding. | Teams use AI-generated mockups/POCs quickly, then refine through rapid feedback loops. | Early outputs can diverge from existing design system if prompts are generic. | Add explicit design-system references in prompts and create an early design checkpoint before implementation scales. |
| Plan and shape | Planning shifts from detailed up-front decomposition to lightweight, rolling plans. | Teams use plan mode to compare approaches and estimate impact (reuse vs new modules, scope of file changes). | Without a near-term plan, teams pick work opportunistically and may miss highest-priority outcomes. | Establish a weekly/biweekly planning cadence with a clearly ordered "next-up" lane. |
| Build and implement | Build speed increases significantly; implementation can happen in parallel across API/UI artifacts. | Agent mode for code generation, small-batch prompting, rapid conversion of specs into code and tests. | Broad prompts trigger large multi-file diffs that are hard to reason about and validate. | Standardize "small-batch implementation" guardrails (limited file scope, explicit constraints, frequent commits). |
| Review feedback | Review changes from occasional milestone reviews to continuous micro-reviews. | Fast prompt->implement->review loops; async clarification across time zones; rapid revision cycles. | If review loops are not scheduled, misalignment persists while velocity continues. | Introduce a lightweight recurring review ritual (daily or alternate-day) tied to active work items. |
| Iterate | Iteration cycles are shorter and more frequent, often happening same day. | Teams quickly re-prompt based on feedback, regenerate artifacts, and test alternatives. | Teams may optimize for speed over coherence, creating fragmented decisions and local fixes. | Capture decision logs per iteration so rationale is preserved and reused across future prompts. |
| QA test & validate | Unit-test generation is faster, but end-to-end confidence still depends on manual QA and integration checks. | AI-assisted unit test authoring for new logic; quick developer-side validation during coding. | Coverage gaps remain for end-to-end behaviors; defects can escape if release confidence is inferred from unit tests alone. | Prioritize automation for highest-risk end-to-end flows and define release-quality gates beyond unit tests. |
| Release & operate in production | Release readiness depends less on coding throughput and more on alignment with external team dependencies. | Teams can rapidly produce integration-ready changes and adjust sequencing in response to partner timelines. | Fast output can outpace cross-team coordination (attribution/reporting/integration), causing timing mismatches. | Use integration-aware release sequencing and shared readiness checkpoints with partner teams. |
| Maintain & debug | Debugging becomes harder when change scope is large or provenance is unclear. | Teams increasingly rely on constrained prompts, smaller diffs, and incremental commits to maintain traceability. | Root-cause analysis slows when AI introduces broad changes across many files. | Enforce traceable change practices: small diffs, explicit prompt intent, and commit-level change narratives. |
| Prioritisation and delivery management | Delivery management shifts from ticket volume to outcome-level prioritization and visibility. | AI can decompose high-level outcomes into detailed execution tasks rapidly; humans operate at higher abstraction. | "Wild west" work pickup, weak backlog structure, and unclear ownership reduce predictability. | Move to dual-layer management: prioritized human-level outcomes plus linked AI execution artifacts/tasks. |

## Cross-stage themes
- **Primary bottleneck has moved:** from coding speed to communication, prioritization, and cross-functional alignment.
- **Context quality is leverage:** better prompt-ready context upstream reduces rework downstream.
- **Process is still required:** AI reduces build friction but increases need for lightweight governance and review cadence.
